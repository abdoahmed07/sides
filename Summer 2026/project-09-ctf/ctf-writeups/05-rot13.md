# PicoCTF — 13 (Cryptography)

**Category:** Cryptography
**Points:** 100

## What the challenge was

A string that appeared to be "encoded" with ROT-13: each letter is shifted by 13. ROT-13 is special because applying it twice gets you back to the original (since 13 + 13 = 26 = full alphabet).

Ciphertext: `cvpbPGS{abg_gbb_onq_bs_n_ceboyrz}`

## How I solved it

ROT-13 is so common it's built into most Unix systems:

```bash
echo "cvpbPGS{abg_gbb_onq_bs_n_ceboyrz}" | tr 'A-Za-z' 'N-ZA-Mn-za-m'
# Output: picoCTF{not_too_bad_of_a_problem}
```

## Encoding vs. Encryption

After solving this I researched the difference:

**Encoding** converts data from one representation to another for compatibility reasons — base64, hex, URL encoding, ROT-13. There's no secret key. Anyone who knows the encoding scheme can decode it.

**Encryption** transforms data using a secret key, and cannot be reversed without that key. AES, RSA, ChaCha20. The algorithm is public; the key is secret.

ROT-13 is encoding, not encryption. It provides zero security because the algorithm itself contains the "key" (shift by 13). The OWASP guide calls this "security through obscurity" — and it doesn't work because obscurity fails as soon as someone knows (or guesses) the scheme.
